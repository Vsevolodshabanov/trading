# План Развития TBank Trader

## 1. Цель системы

Построить модульную авто-трейдинг платформу на базе `T-Invest API`, которая:

- торгует в `sandbox` и затем в ограниченном `paper/live` режиме;
- поддерживает несколько классов стратегий: `momentum`, `mean reversion`, `news-driven`, `relative value`;
- имеет отдельный слой исполнения, риска, наблюдаемости и UI;
- допускает подключение reasoning/LLM-компонента вне горячего цикла исполнения.

## 2. Текущее состояние

На текущий момент уже есть:

- Docker-стек: `api`, `engine`, `db`, `redis`;
- FastAPI dashboard и API;
- рабочий `sandbox`-адаптер T-Bank;
- загрузка последних цен и исторических свечей;
- warmup стратегий на старте;
- стратегии:
  - momentum MA-cross для `shares/fx`;
  - mean reversion для `bonds`;
- policy-слой исполнения:
  - отдельные лимиты для `shares / bonds / fx`;
  - inventory-aware filter для long-only инструментов;
- risk-слой:
  - ограничения по количеству;
  - ограничения по `RUB notional`;
- синхронизация позиций и журнал сигналов/ордеров;
- фронт с объяснением стратегий и execution guardrails.

## 3. Архитектурные принципы

Систему стоит развивать по следующим правилам:

1. Горячий торговый цикл должен быть детерминированным.
   В нем не должно быть медленных и недетерминированных LLM-вызовов.

2. Reasoning-компонент должен быть асинхронным.
   Его задача: интерпретация новостей, режим рынка, пост-анализ, генерация гипотез.

3. Execution и Risk должны быть отдельными слоями.
   Стратегия не должна сама решать вопросы лотов, лимитов, шортов и kill-switch.

4. Каждая стратегия должна проходить путь:
   `research -> backtest -> sandbox -> restricted production`.

5. UI должен быть не только мониторинговым, но и диагностическим.
   Пользователь должен понимать, почему сделка была совершена или отклонена.

## 4. Целевая архитектура

Итоговая схема:

- `market-data worker`
  - получает котировки, свечи, стаканы, торговые статусы, новости;
  - пишет краткоживущие данные в `Redis`;
  - пишет историю и события в `Postgres`;

- `strategy engine`
  - читает нормализованный market state;
  - генерирует сигналы;
  - не занимается напрямую лимитами и брокерскими исключениями;

- `execution policy`
  - определяет размер ордера;
  - применяет asset-class rules;
  - проверяет inventory constraints;

- `risk engine`
  - применяет symbol limits, notional limits, daily loss limits, kill-switches;
  - может учитывать advisory state от reasoning worker;

- `broker adapter`
  - отправляет ордера;
  - подтверждает статусы;
  - синхронизирует позиции и денежные остатки;

- `reasoning worker`
  - обрабатывает новости, режим рынка, аномалии;
  - не ставит ордера напрямую;
  - пишет advisory state и explanations;

- `api/ui`
  - показывает состояние системы;
  - позволяет включать/выключать движок;
  - объясняет решения и отклонения.

## 5. Очередность разработки

### Фаза A. Стабилизация sandbox-контура

Цель:
довести текущую систему до надежного sandbox-контура с предсказуемым поведением.

Задачи:

- добавить торговые статусы инструментов перед отправкой ордеров;
- добавить recheck на возможность торговли для `FX` и иных классов;
- добавить защиту от repeated broker errors по символу;
- хранить `broker_error` с кодом, текстом и категорией;
- ввести `cooldown` после ошибки исполнения;
- записывать больше execution metadata в БД.

Критерий завершения:

- система несколько дней работает в sandbox без ручных правок;
- ошибки исполнения объяснимы и не повторяются бесконтрольно.

### Фаза B. Улучшение market data слоя

Цель:
перейти от простого polling к более надежному market state.

Задачи:

- добавить загрузку краткой истории свечей по расписанию;
- ввести свежесть данных по каждому символу;
- добавить торговые статусы и признаки доступности инструмента;
- подготовить слой для потоковых данных;
- ввести локальный `market snapshot` объект для стратегии.

Критерий завершения:

- стратегия получает унифицированный state, а не только last price.

### Фаза C. Research и backtesting

Цель:
получить честную экспериментальную среду.

Задачи:

- реализовать офлайн backtest runner;
- загрузить исторические свечи в локальное хранилище;
- поддержать конфиги экспериментов в `yaml/json`;
- считать:
  - pnl;
  - max drawdown;
  - hit ratio;
  - average holding time;
  - turnover;
  - proxy slippage;
- сохранять отчеты прогона в `docs/experiments` или отдельную таблицу.

Критерий завершения:

- любая новая стратегия запускается как воспроизводимый эксперимент.

### Фаза D. Расширение стратегий

Цель:
увеличить число осмысленных торговых моделей.

#### 1. Акции

- intraday momentum breakout;
- continuation after volume expansion;
- short-term reversal after exhaustion;
- session-aware logic;
- spread/liquidity filters.

#### 2. Облигации

- mean reversion на rolling fair value;
- liquidity-aware exits;
- bucket rules по duration;
- отдельные лимиты на низколиквидные инструменты.

#### 3. FX

- momentum with session filters;
- response to cross-asset confirmation;
- tighter order notional policy;
- отдельные execution rules по торговой доступности.

#### 4. Relative value / quasi-arbitrage

- акции против секторальных прокси;
- облигации внутри одного сегмента;
- валютные спрэды и кросс-подтверждения;
- стат-арбитраж допускается только после нормального data/research контура.

Критерий завершения:

- есть не одна “демо”-стратегия, а портфель независимых гипотез.

### Фаза E. News и reasoning layer

Цель:
встроить LLM осмысленно, а не в горячий loop каждые 5 секунд.

Подход:

- новости поступают отдельным worker;
- reasoning worker делает:
  - краткое резюме новости;
  - классификацию: positive / negative / neutral / uncertain;
  - оценку релевантности к торгуемому universe;
  - оценку временного горизонта реакции;
  - advisory flag для risk/strategy;
- execution loop читает уже готовый advisory state.

Что не надо делать:

- не вызывать reasoning model на каждый тик;
- не позволять LLM напрямую отправлять ордер;
- не принимать решения без risk gate и deterministic checks.

Первые кейсы для reasoning worker:

- разбор headline/news event;
- определение risk-off / risk-on режима;
- пост-мортем по необычным сериям broker_error;
- генерация гипотез для следующих экспериментов.

Критерий завершения:

- reasoning используется как фильтр и аналитик, а не как “магический трейдер”.

### Фаза F. Production-grade controls

Цель:
подготовить систему к ограниченному реальному использованию.

Задачи:

- daily loss limits;
- max open exposure;
- max simultaneous positions;
- kill switch;
- cooldown after drawdown or repeated errors;
- heartbeat monitoring и алертинг;
- audit trail всех решений;
- secret management;
- конфигурации окружений `dev / sandbox / restricted prod`.

Критерий завершения:

- систему можно безопасно запускать без постоянного ручного надзора.

## 6. План по фронту

Фронт должен эволюционировать от “dashboard” к “trading cockpit”.

Следующие UI-шаги:

- история broker errors и исполнений;
- карточки по стратегиям с текущим состоянием;
- timeline событий;
- фильтрация по символам/стратегиям;
- раскрытие объяснения сигнала:
  - signal;
  - sizing;
  - risk;
  - broker result;
- pnl и exposure по классам активов;
- блок market regime / advisory state;
- страница экспериментов и отчетов.

## 7. Набор ближайших практических задач

### Ближайшая итерация

- добавить `trading status` checks перед ордером;
- добавить symbol cooldown после broker error;
- улучшить обработку `broker_error` для `FX`;
- вывести на фронт breakdown причин отклонения;
- добавить asset-class exposure summary;
- сделать отдельный execution policy monitor.

### Следом

- сделать backtest runner;
- завести конфиги экспериментов;
- подготовить historical data store;
- добавить вторую акционную стратегию;
- добавить bond exit management;
- начать отдельный news worker.

## 8. Критерии качества стратегии

Каждая стратегия должна оцениваться минимум по:

- expected value;
- drawdown;
- turnover;
- slippage sensitivity;
- доле отклоненных ордеров;
- стабильности между днями;
- explainability на уровне signal -> execution -> result.

## 9. Правило допуска стратегии дальше

Стратегия переходит на следующий этап только если:

- прошла backtest;
- прошла sandbox без системных сбоев;
- показывает положительную ожиданиемость после комиссий и slippage assumptions;
- не нарушает риск-лимиты;
- имеет понятный operational profile.

## 10. Приоритет на ближайшее время

Самый рациональный фокус сейчас:

1. стабилизировать execution layer;
2. добавить backtest/research контур;
3. только потом расширять news/reasoning и сложные cross-asset идеи.

Иначе система слишком рано станет сложной и плохо диагностируемой.
